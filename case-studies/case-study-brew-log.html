Brew Log: A Homebrewing Batch Tracker

I homebrew mead. I wanted a more focused tracker than Google Docs, but didn't want to pay for it. So I built one.

Type Personal Project
Stack Vanilla JS (ES modules), localStorage, IndexedDB
Status Live
LIVE PROJECT Open Brew Log → View source on GitHub →

How It Started

I homebrew mead. For a while I tracked batches in Gdocs, which... worked... but quickly grew out of control and sloppy as my notes merged in with the instructions.

This didn't start as a product idea. I wasn't trying to build something for other brewers. I just wanted a tool that fit the actual workflow of making a batch — recipe first, then tracking, then notes — without forcing me to switch between different files or apps.

What I Built

Each batch gets a card with a photo thumbnail, a recipe editor with an ingredients list and step-by-step instructions (markdown supported), and a gravity log that estimates ABV as you add readings. Tasting notes have attribute tags so you can characterize a batch in a consistent way across batches without free-form text getting hard to compare.

Brew mode turns the recipe steps into a sequential checklist. When you work through the last step, the batch auto-marks as "It's done!" — small thing, but satisfying in practice.

The architecture is localStorage for structured metadata and IndexedDB for photos, keeping everything on-device. There's no backend and no account. A full JSON export/import lets you back up or move data. Recipes can also be shared as a URL with the recipe encoded as a base64 hash — no server needed, just a link.

A Few Design Decisions

The most interesting one was the save model. I started with autosave, which seemed like the obvious choice — every change commits automatically, nothing gets lost. But autosave was giving me headaches (fire too often and it was losing focus on the text field, for example). Better to keep it simple, so I went with a manual save + notifier if you close the app with unsaved changes.

The other notable moment was using IndexedDB for the first time. localStorage's 5MB cap makes it the wrong choice for images — something I'd been aware of in the abstract. IndexedDB's API is verbose and async in a way that took some getting used to, but once I understood the pattern the split worked cleanly. Structured data in one place, blobs in another, both local, both fast.

What I'd Do Differently

The shareable recipe links were added late, as a nice-to-have. They work, but they're bolted on rather than designed in — the URL is longer than it needs to be because the entire recipe object gets encoded rather than a leaner representation. If I'd thought about sharing earlier I'd have built a minimal data model for it from the start. Not a real problem in practice, but a clear sign of where the architecture was added versus planned.

I'd also think harder upfront about which features actually need to sync across devices and which don't. The export/import flow is fine for backups, but there are moments — like wanting to reference a recipe from my phone while the batch is running — where a simple read-only URL per batch would have been more useful than a full JSON export.

All said, it gets the job done and looks decent. Go ahead and try it -- just remember that the data is stored locally, so make sure to save, import, and export accordingly.
