JD Sifter: An AI-Powered JD Evaluator

Built during a job search to replace a tedious manual workflow. The problem wasn't generating cover letters — it was figuring out which of 15 saved jobs were even worth pursuing.

Type Personal Project
Stack Vanilla JS, Anthropic API
Status Live, actively used
Method AI-assisted prototyping
LIVE PROJECT Open JD Sifter →

Context

I was actively job searching and had developed a reasonable workflow: save interesting roles on LinkedIn, then evaluate them one by one in a Claude chat thread with my background and preferences loaded as context. The AI evaluation was genuinely useful — the bottleneck was everything before it.

For each job, the process was: open the listing, select all, copy, switch to the chat tab, paste, wait, read the response, switch back to LinkedIn, repeat. For two or three jobs, fine. For fifteen, it's a grind — and the cognitive switching between tabs and chat threads added friction that meant I'd often skip the evaluation step entirely and just eyeball it.

Problem

The right framing took a conversation to arrive at. My first instinct was to build something that could scrape job boards and auto-surface matches — the "crawl LinkedIn once a day" vision. That's a real problem, but it's the wrong one to solve first: job boards actively block scrapers, the interesting API integrations require significant infrastructure, and the tool would need to exist and prove value before that investment made sense.

The actual constraint was simpler. I already had a good evaluation tool — a chat thread with a detailed profile prompt. The bottleneck was the intake step, not the analysis step. Each evaluation required switching to a separate tab, copying raw text, and pasting it into a separate conversation. Fifteen jobs meant fifteen round trips. What I needed was a triage layer that could sit in front of the existing workflow: load the jobs in bulk, screen them fast, hand off the winners to the existing process.

What I Built

A single HTML file that connects to the Anthropic API directly from the browser. No server, no hosting cost beyond the API calls themselves — fractions of a cent per job at Sonnet pricing.

The core interface is a column of paste slots. The UX is optimized for a specific workflow: have ten LinkedIn tabs open, alt-tab to the app, ctrl+V (the full text dump from a job page), auto-advance to the next slot, alt-tab back to LinkedIn, close the tab, repeat. With paste-advance enabled, loading fifteen jobs takes about thirty seconds. Hit Evaluate All and the API calls fire in parallel — all slots evaluated simultaneously rather than sequentially.

Each result card shows a verdict (Apply / Maybe / Skip), a fit score, a headline reason, green flags, red flags, and a short paragraph of reasoning. The scoring is grounded in a detailed profile: target role types, dealbreakers, comp floor, remote requirements, domain preferences, what I'm moving toward and away from. The profile is editable in-app and persists to localStorage, so it doesn't have to be copy-pasted into every new session.

A saved applications list lets you track which jobs you've already acted on — useful when the same company posts the same role across multiple job boards, or when you want to note "applied via LinkedIn" before closing the tab for good. Saved entries include a free-form notes field for exactly this kind of annotation.

Key Decisions

Paste-and-advance over a single text dump. An early idea was to support pasting all JDs at once, separated by a delimiter. The problem is that a raw LinkedIn copy includes navigation chrome, sidebar content, and footer noise — splitting a combined blob reliably is brittle. Individual slots with auto-advance keep the per-job separation clean while still minimizing friction in the intake flow.

Parallel evaluation, not sequential. Evaluating slots one by one would mean waiting for each response before the next fires. At 3–5 seconds per call, fifteen jobs would take over a minute. Firing all API calls simultaneously means the whole batch returns in roughly the time of a single evaluation. The loading state shows individual spinners per card so you can see results as they arrive.

Editable profile over a hardcoded one. The original build had the profile baked in. That became a problem quickly — a job search isn't static. Early sessions might prioritize AI-native products; later, you might open the criteria to include adjacent domains. Rebuilding the file every time the criteria shifted was friction that would eventually make the tool feel like a liability. Storing the profile in localStorage with an in-app editor means it evolves with the search.

Standalone HTML file. The format matches the rest of this portfolio's tooling philosophy: no install, no login, no subscription. You open a file. The API key lives in localStorage, entered once. Everything works offline except the evaluation itself.

Outcome

The triage layer now sits cleanly in front of the existing chat workflow. Load fifteen jobs in thirty seconds, get results in another ten, carry the three or four that score above 70 into the cover letter and Q&A process. The rest get archived with a note about why they didn't make the cut.

The more durable outcome is that the profile itself — the scoring rubric that lives inside the tool — is now a maintained artifact rather than context buried in a chat thread. Chat threads drift. A saved profile document doesn't.
