Personal Project · 2024–Present

From Friction to Feature Roadmap: Building a Music Analysis Tool

A narrow personal pain point became a prototyping vehicle for something bigger — and the process of building it turned out to be as interesting as the tool itself.

Type Personal Project
Stack Vanilla JS, Web Audio API, Meyda.js
Status Working prototype, user testing
Method AI-assisted prototyping (vibe coding)

Context

This is a personal project I built to scratch my own itch, using AI-assisted development as a deliberate prototyping method. I play piano and produce music in Ableton. I am not a developer, but I wanted to see how far I could get with a genuine problem and a plan-first approach.

Problem

When I jam on piano and land on something I want to develop into a track, my first step is figuring out the BPM of what I just played. It sounds simple. In practice, it's clunky: export from Ableton, try to tap it out, guess and check. No existing tool handled short, unfinished loops well. Most BPM detectors assume a full song, not a four-bar idea you just recorded.

That was the original problem. The more interesting one surfaced later.

What I Did

Before writing any code, I described the problem to Claude and asked whether it was actually tractable — feasibility first, building second. That conversation shaped the scope significantly. One early idea was full audio-to-MIDI transcription, which would have let me recreate tracks produced in tools like Suno directly in Ableton. After talking through the landscape, I shelved it: perfect transcription is an unsolved ML problem, the existing tools all require too much cleanup, and the "audio forensics" layer (instrument identification, effects detection) needed a different tech stack entirely. My existing workflow was already near-optimal for that use case. Knowing what not to build freed me up to focus on what I could actually ship.

The other deliberate decision was the format. The output is a single HTML file — no server, no hosting cost, no install friction. That was a product decision, not a technical limitation. Keeping it as frictionless as possible to open and use was the point. A web app with a login screen would have killed the use case for me entirely.

What shipped first was straightforward: BPM detection via audio file upload, tap tempo with a rolling average, and waveform visualization with a beat grid overlay. It worked well. And then the interesting part happened.

Once I had something real to interact with, the problem space opened up. I started using Moises and Fadr to separate stems from tracks I wanted to recreate, and realized I could drop those stems into the tool individually and extract useful information about each one — key, dominant pitch class, timbre, energy profile. That became the stem analysis tab: a feature designed around a specific use case (recreating a Suno-generated track in Ableton) that only became visible because I had a prototype to think with. None of that was in the original brief.

The roadmap that emerged through prototyping includes mic recording to remove the Ableton export step entirely, a tempo drift map to chart BPM variation across a full recording, a session logger with trend analysis across practice sessions (a potential monetization gate), and a groove quantifier that measures where your hits land relative to the grid. These weren't planned features — they were discovered ones.

Outcome

The prototype works well and solves the original problem. I'm currently testing it with a small group of musician friends to see whether the friction points I felt are shared, and whether the stem analysis feature lands the way I think it does. Productizing it is a real possibility; nothing decided yet.

The more immediate outcome is a clearer sense of the opportunity. What started as a BPM detector is now a focused music analysis tool with a coherent feature roadmap, a specific target user (producers who work with a DAW and want to understand what they're hearing), and at least one feature that doesn't exist anywhere else in quite this form.

What I Learned

Scoping instinct is a real skill. I pushed back toward simplicity several times during this project — single file, no backend, scope the stem analysis down rather than up. Each of those calls made the prototype more useful, not less. There's a version of this project I could have spent three months on and never shipped.

Vibe coding is actually a discovery method. I used AI-assisted development deliberately to find out what the problem actually was, not just to generate code faster. The BPM finder surfaced the stem analysis idea, which surfaced the Suno recreation use case — none of that was visible from the outside. The prototype was the research.

Knowing when to shelve matters as much as knowing what to build. The transcription idea was interesting and I spent real time on it. Letting it go cleanly, with a documented rationale, was the right move. I've tried to apply that same instinct here: the roadmap is real, but I'm not building all of it yet.

Plan mode first. Every time I sat down to build something new on this project, I started with a scoping conversation before writing any code. That habit kept the scope from drifting and made the builds faster when they happened. It's the same instinct I try to bring to work projects — just applied to a side project where nobody was asking me to slow down.
